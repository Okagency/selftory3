 <h4>10.3 PP 모듈 인터뷰 표준 질문 35개 — 생산 (시니어의 진짜 답)</h4>

 <p>각 질문에 — 왜 묻는가 / 진짜 신호 / 어떻게 검증하나 / 다음 질문 / 프로젝트 영향 5단으로 적었다. 네이버에 안 나오는 자리.</p>

 <ol>
 <li><strong>한성정밀 주요 제품군은 몇 가지이며, 양산형/주문형 비중은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: MTS·MTO·ETO 시나리오 비중을 결정하기 위해. 비중이 다르면 SD-PP 통합 시나리오 매트릭스 자체가 달라진다.</li>
 <li><strong>진짜 신호</strong>: 양산형 80% + 주문형 20%로 답하면 — 보통 거짓말이다. 실제로는 양산형 50% + 주문형 50%인데 영업이 "우리는 양산이 주력"이라고 자존심으로 답한다.</li>
 <li><strong>어떻게 검증하나</strong>: 작년 SO 데이터에서 '단일 고객 단일 품목 1회 수주' 라인을 직접 세어 분류한다. 시스템 답이 아니라 — Raw 데이터 답을 받는다.</li>
 <li><strong>다음 질문</strong>: "최근 1년 SO 라인을 한 번 같이 봐도 될까요? 양산·주문 분류를 직접 검증해 보고 싶어서요."</li>
 <li><strong>프로젝트 영향</strong>: 잘못 설계 시 — UAT 단계에서 "이 시나리오로는 우리 일이 안 된다" 폭발. 재설계 6~8주, 추가 비용 4~6억.</li>
 </ul></li>

 <li><strong>월 평균 작업오더 발행 건수는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 시스템 부하·라이선스·MRP 사이클 산정. 그러나 진짜 의도는 — 작업오더 1건당 BOM 라인 수의 분포를 알아내기 위함이다.</li>
 <li><strong>진짜 신호</strong>: 작업오더 건수 자체보다 — '평균 작업오더 1건당 BOM 라인 수'가 진짜 신호. 5라인 = 단순. 50라인 이상 = 다단계 BOM, 외주가공·반제품 복잡도 폭발.</li>
 <li><strong>어떻게 검증하나</strong>: 작년 작업오더 100건 샘플을 뽑아 — BOM 라인 수 분포(중앙값·90퍼센타일)를 직접 측정.</li>
 <li><strong>다음 질문</strong>: "월말에 작업오더 미종결(WIP) 평균 잔여 건수는요? 90일 이상 미종결은 몇 % 인가요?"</li>
 <li><strong>프로젝트 영향</strong>: WIP 잔여가 발행 건수의 30% 이상이면 결산 시 재공품 평가 정합성 깨짐. D+5 결산 가능성과 직결.</li>
 </ul></li>

 <li><strong>생산방식은 (MTS·MTO·ETO·ATO) 어느 것이 주축?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: SD-PP 통합 시나리오의 첫 분기점. 한 가지 답으로는 시나리오 설계가 끝나지 않는다.</li>
 <li><strong>진짜 신호</strong>: 답이 한 가지로 깔끔하게 나오면 — 의심하라. 한국 1차 협력사의 80%는 '고객별로 방식이 다르다'가 진짜 답. H자동차는 MTO, K자동차는 MTS, 해외 OEM은 ETO.</li>
 <li><strong>어떻게 검증하나</strong>: 매출 상위 5곳 고객의 — 발주 패턴(JIS·내시·확정·예측) 자료를 받아 분류.</li>
 <li><strong>다음 질문</strong>: "주요 고객 5곳 각각의 발주 패턴을 한 번 같이 봐도 될까요? 시나리오를 고객별로 분리할지 결정해야 해서요."</li>
 <li><strong>프로젝트 영향</strong>: 시나리오 분리 = SD-PP 통합 매트릭스 5×3 = 15개 시나리오 신설. 시나리오 단일 설계 = 한 고객에서 무너짐.</li>
 </ul></li>

 <li><strong>BOM 다단 계층은 평균 몇 단계? 최대는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: MRP 폭발(Explosion) 시간·Phantom 사용 비중·반제품 운영 정책 결정.</li>
 <li><strong>진짜 신호</strong>: 평균 3단계 / 최대 7단계가 답이면 정상. 그러나 '최대 12단계 이상'이 나오면 — Phantom Assembly로 분류된 자리가 있다는 뜻. Phantom 비중 30% 이상이면 MRP 폭발 시간이 평소의 3배.</li>
 <li><strong>어떻게 검증하나</strong>: 현재 ERP에서 — BOM 단수 분포 리포트를 추출(SAP는 CS12). 단수별 자재 수와 Phantom 비중을 본다.</li>
 <li><strong>다음 질문</strong>: "Phantom Assembly로 잡힌 BOM이 몇 % 인가요? 그 자리들 한 번 같이 봐도 될까요?"</li>
 <li><strong>프로젝트 영향</strong>: Phantom 처리 잘못하면 — 원가 차이 분석 시 Quantity Variance 폭발. CO에서 풀 수 없음.</li>
 </ul></li>

 <li><strong>BOM 등록·변경은 누가, 어떤 승인 절차로?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: BOM 정합성이 컷오버 D-30의 가장 큰 변수. 정합성 70% 이하면 컷오버 12주 추가.</li>
 <li><strong>진짜 신호</strong>: "설계팀이 등록, 생산팀이 승인"이 답이면 — 표면 답. 실제로 물어야 할 것은 '긴급 변경(Emergency BOM Change) 시 승인 우회 빈도'. 월 5건 이상 = 정합성 70% 이하.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 1년 BOM 변경 이력을 받아 — 정상 승인 vs 긴급 우회 비율 측정. 우회 사유 카테고리 분류.</li>
 <li><strong>다음 질문</strong>: "지난 1년 긴급 BOM 변경 건수와 사유 카테고리 자료를 받을 수 있을까요?"</li>
 <li><strong>프로젝트 영향</strong>: 정합성 70% 이하 = 컷오버 마스터 정제 12주 추가. 정합성 90% 이상 = 컷오버 D-30 안정.</li>
 </ul></li>

 <li><strong>설계변경(ECO)은 연 몇 건이며, 평균 처리 기간은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: ECM(Engineering Change Mgmt) 모듈 활성화 결정. ECO 적용 일자(Effectivity Date) 운영 정합성 측정.</li>
 <li><strong>진짜 신호</strong>: 연 ECO 200건 + 평균 처리 14일은 표면 답. 진짜 신호는 — 'ECO 발의 → 양산 적용까지 실제 사이클타임'. 60일 이상이면 적용·미적용 변경분이 BOM에 혼재.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 6개월 ECO 10건을 — 발의일·승인일·BOM 반영일·실제 양산 적용일 4단계로 측정.</li>
 <li><strong>다음 질문</strong>: "ECO 적용 일자(Effectivity Date) 관리 — From/To 일자를 BOM에 직접 박나요, 별도 테이블인가요?"</li>
 <li><strong>프로젝트 영향</strong>: ECO 정합성 = SAP ECM 모듈 활성화 결정. 24장 ECO 워크플로 핵심 입력값.</li>
 </ul></li>

 <li><strong>EBOM과 MBOM이 분리되어 있는가? 통합인가?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: PLM-ERP 인터페이스 견적의 핵심 정당화 자료. EBOM-MBOM 자동화가 ERP 도입의 가장 큰 가치 제안인 자리가 많다.</li>
 <li><strong>진짜 신호</strong>: "통합"이라고 답하면 — 80%는 거짓말. 실제로는 설계팀의 EBOM 엑셀과 생산팀의 MBOM ERP가 따로 굴러가고, 사람이 매주 수기 동기화.</li>
 <li><strong>어떻게 검증하나</strong>: PLM 시스템 화면에서 BOM 1건을 뽑고, ERP 화면에서 같은 자재의 BOM을 뽑아 — 라인별로 일치 여부를 직접 비교한다.</li>
 <li><strong>다음 질문</strong>: "PLM의 EBOM과 ERP의 MBOM을 동기화하는 사람이 있나요? 매주 몇 시간 걸리시나요?"</li>
 <li><strong>프로젝트 영향</strong>: EBOM-MBOM 자동화 = PLM-ERP 인터페이스 견적 5~10억의 핵심 정당화. 60장 PLM-ERP 통합 입력값.</li>
 </ul></li>

 <li><strong>라우팅은 표준화되어 있는가? 작업장별 표준 시간은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 표준원가 정확도와 직결. 표준원가-실제원가 차이가 매년 누적되는 원인이 — 갱신 안 된 표준 시간인 경우가 많다.</li>
 <li><strong>진짜 신호</strong>: "표준화돼 있다"가 답이면 — Standard Time이 언제 측정됐는지 물어라. 5년 이상 안 갱신된 회사가 60%.</li>
 <li><strong>어떻게 검증하나</strong>: 라우팅 마스터에서 표준시간 — 마지막 수정일자(SAP는 PLPO 테이블 변경일) 분포를 본다. 5년 이상 비중을 측정.</li>
 <li><strong>다음 질문</strong>: "표준 시간 마지막 측정 일자가 언제인가요? 작업 방법(IE)이 그 사이 바뀌었나요?"</li>
 <li><strong>프로젝트 영향</strong>: 표준 시간 재측정 4~6주 추가. 안 하면 표준원가 신뢰도 0. 51.0 표준원가 결산과 직결.</li>
 </ul></li>

 <li><strong>워크센터(작업장) 구성과 능력 산정 방식은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: CRP(Capacity Requirements Planning) 의미 있게 만들기 위해. 병목 식별이 안 되면 CRP는 그림에 불과.</li>
 <li><strong>진짜 신호</strong>: "라인별 1개 워크센터"가 답이면 — 능력 산정 부정확. 같은 라인에 병목 1개 + 비병목 5개가 섞여 있어서 워크센터 구분이 필요.</li>
 <li><strong>어떻게 검증하나</strong>: 라인별로 — 가동률·지연 발생 빈도 데이터를 받아 병목 공정을 식별한다. TOC(Theory of Constraints) 적용 가능 여부 측정.</li>
 <li><strong>다음 질문</strong>: "라인 안에서 가장 자주 멈추는 공정 1~2개를 식별해 두셨나요? Bottleneck이라는 단어로 운영하시나요?"</li>
 <li><strong>프로젝트 영향</strong>: 병목 식별 = TOC 적용 = 생산 처리량 15~30% 향상 가능. CRP 안정성과 직결.</li>
 </ul></li>

 <li><strong>MRP 사이클은 일/주/월 단위?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 인프라 부하·자동화 수준·운영 성숙도 측정.</li>
 <li><strong>진짜 신호</strong>: "주 1회"가 답이면 — 정상. "일 1회"가 답이면 — Net Change MRP 운영, 인프라 부하 큼. "월 1회"가 답이면 — 안전재고로 버티는 회사, 데드 스톡 누적.</li>
 <li><strong>어떻게 검증하나</strong>: MRP 결과 PR(구매요청) 자동 변환율을 측정. 자동 80% 이상 = 성숙. 50% 이하 = 수동 검토 부담.</li>
 <li><strong>다음 질문</strong>: "MRP 결과를 누가 검토하고 PR로 확정하나요? 자동인가 수동인가? 자동률은 얼마인가요?"</li>
 <li><strong>프로젝트 영향</strong>: 자동 PR 변환율 = ERP 가치 측정의 핵심 KPI.</li>
 </ul></li>

 <li><strong>Lot Size 정책은? (LFL·EOQ·POQ·고정)</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 운전자본 절감 ROI 산정. 자재별 차등 적용 가능성 측정.</li>
 <li><strong>진짜 신호</strong>: "고정 Lot"이 답이면 — 90% 회사. 그러나 자재별로 EOQ 적용해야 할 자재가 30%는 섞여 있다. 일괄 정책으로는 운전자본 5~10% 손실.</li>
 <li><strong>어떻게 검증하나</strong>: 자재별 — 셋업 비용·재고 보관 비용 데이터를 받아 EOQ 공식으로 재산정 시뮬레이션. 차이 자재 비중 측정.</li>
 <li><strong>다음 질문</strong>: "자재별로 셋업 비용·재고 보관 비용이 다른데, 모두 같은 Lot Size를 쓰나요? EOQ 적용 검토해 본 적 있나요?"</li>
 <li><strong>프로젝트 영향</strong>: Lot Size 차등 적용 = 운전자본 절감 매년 수억.</li>
 </ul></li>

 <li><strong>안전재고 정책과 산정 근거는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 운전자본 대규모 절감 가능성 측정. 안전재고 25% 감소 가능 자리가 많다.</li>
 <li><strong>진짜 신호</strong>: "경험으로 정함"이 답이면 — 100% 정상. 그러나 수요 변동의 표준편차 × 서비스레벨 Z값 × √리드타임 공식으로 재산정하면 — 평균 안전재고 25% 감소 가능.</li>
 <li><strong>어떻게 검증하나</strong>: 자재 10건 샘플로 — 주별 수요 표준편차를 계산하고 공식 적용. 현재 안전재고 vs 공식 안전재고 비교.</li>
 <li><strong>다음 질문</strong>: "수요 변동성 데이터(주별·월별 표준편차)를 가지고 계신가요? 안전재고 공식 기반 재산정에 동의하시나요?"</li>
 <li><strong>프로젝트 영향</strong>: 안전재고 재산정 별도 견적 1~2억. 운전자본 매년 5~10억 절감.</li>
 </ul></li>

 <li><strong>외주공정 비중과 외주 자재 사급 방식은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: SAP Subcontracting BOM 운영 필요성·MM-PP 통합 시나리오 결정.</li>
 <li><strong>진짜 신호</strong>: 외주 비중 30% 이상 + 사급재 추적 수기 = SAP Subcontracting BOM 미운영. 매월 사급재 재고 정합성 깨짐. 외주업체 청구서와 자사 출고 기록 매번 불일치.</li>
 <li><strong>어떻게 검증하나</strong>: 외주업체 1곳 선정해 — 우리 출고 기록과 외주업체 잔여 재고 보고서를 직접 대조. 차이 비중 측정.</li>
 <li><strong>다음 질문</strong>: "외주업체에 사급한 자재의 월말 잔여 재고는 어떻게 확인하시나요? 외주업체 시스템과 우리 시스템이 동기화되나요?"</li>
 <li><strong>프로젝트 영향</strong>: Subcontracting BOM 자동화 = MM-PP 통합 시나리오 핵심.</li>
 </ul></li>

 <li><strong>생산실적 집계는 자동(MES)/수동/혼합?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 32장 MES 인터페이스 안정성 = 컷오버 D-1 War Room의 핵심 점검 항목.</li>
 <li><strong>진짜 신호</strong>: "MES 자동"이 답이면 — MES-ERP 인터페이스 안정성과 시차를 물어라. 실시간 vs 배치(시간 단위) vs 일 1회가 결산 D+5 가능성과 직결.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 30일 MES → ERP 송신 — 실패 건수, 재처리 건수, 사람이 수기 보정한 건수를 측정.</li>
 <li><strong>다음 질문</strong>: "MES → ERP 송신 시 실패율이 어느 정도인가요? 실패 시 재처리 절차는? 수기 보정은 한 달에 몇 건인가요?"</li>
 <li><strong>프로젝트 영향</strong>: MES 송신 실패율 1% 이상 = 결산 D+5 위험. 인터페이스 재설계 검토.</li>
 </ul></li>

 <li><strong>실적 집계 주기는? (시간 단위·교대 단위·일 단위)</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: OEE(Overall Equipment Effectiveness) 측정 가능성·실시간 가시성 수준 측정.</li>
 <li><strong>진짜 신호</strong>: 일 단위가 답이면 — 시간별 OEE 측정 불가. 교대 단위 이상이어야 OEE 의미 있음.</li>
 <li><strong>어떻게 검증하나</strong>: 현재 OEE 보고서를 받아 — 측정 주기·계산 공식·책임자를 확인. 가용성·성능·품질 3요소 분리 여부 본다.</li>
 <li><strong>다음 질문</strong>: "OEE를 측정하시나요? 측정 주기와 책임자는? 가용성·성능·품질 3요소로 분해하시나요?"</li>
 <li><strong>프로젝트 영향</strong>: OEE 자동화 = MES + Fiori 대시보드 신설. 라인별 OEE 실시간 가시화.</li>
 </ul></li>

 <li><strong>불량 처리 프로세스 — 부적합·재작업·폐기 구분은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 품질 비용(COQ) 분석 가능성·QM-CO 통합 시나리오 결정.</li>
 <li><strong>진짜 신호</strong>: 구분 명문화 안 됨 = 불량 비용이 매출원가에 그냥 흡수. 진짜 불량률 모르는 회사. COQ 분석 불가.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 분기 불량 건 10건을 — 재작업·폐기·고객 반품으로 분류. 비용 분개 자동 vs 수기 비율 측정.</li>
 <li><strong>다음 질문</strong>: "불량 → 재작업 vs 폐기 결정을 누가 하나요? 재작업 횟수 한계는? 폐기 비용은 어느 계정에 떨어지나요?"</li>
 <li><strong>프로젝트 영향</strong>: 불량 분개 자동화 = QM-CO 통합. 품질 투자 의사결정 근거 자동 산출.</li>
 </ul></li>

 <li><strong>로스율(Scrap %) 관리 — 표준 vs 실제 차이는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 표준원가 신뢰도와 직결. Scrap Factor가 BOM에 박혔는지가 첫 점검.</li>
 <li><strong>진짜 신호</strong>: 표준 Scrap이 BOM에 박혀 있나요? 안 박혔으면 — 차이분석 시 Quantity Variance 끊임없이 발생.</li>
 <li><strong>어떻게 검증하나</strong>: BOM 마스터에서 Scrap Factor 필드 — 입력 비중·갱신 주기 측정. 자재별 실제 Scrap %와 표준 Scrap % 비교.</li>
 <li><strong>다음 질문</strong>: "BOM의 Scrap Factor 필드를 쓰고 계신가요? 표준 Scrap %는 누가 정하나요? 마지막 갱신 일자는?"</li>
 <li><strong>프로젝트 영향</strong>: Scrap Factor 정확화 = 표준원가 신뢰도 직결.</li>
 </ul></li>

 <li><strong>작업오더 종결(Confirmation) 시점·기준은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: WIP 평가 정확도·D+5 결산 가능성과 직결.</li>
 <li><strong>진짜 신호</strong>: 작업오더가 90일 이상 미종결로 남는 비율 — 30% 이상이면 결산 정합성 무너짐.</li>
 <li><strong>어떻게 검증하나</strong>: 현재 ERP에서 — 작업오더 미종결 일자 분포 리포트 추출. 30·60·90·180일 미종결 비중 측정.</li>
 <li><strong>다음 질문</strong>: "월말에 미종결 오더가 평균 몇 건이고, 처리 책임자는 누구인가요? 90일 이상은 누가 책임지나요?"</li>
 <li><strong>프로젝트 영향</strong>: 미종결 오더 처리 자동화 = D+5 결산 가능성.</li>
 </ul></li>

 <li><strong>표준원가 vs 실제원가 차이는 평균 몇 %?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: K-IFRS 1002호 '실제와 유사한 경우' 조건 충족 여부. 외부감사 지적 위험 측정.</li>
 <li><strong>진짜 신호</strong>: ±5% 이내 = 표준원가 결산 OK. 7% 이상 = 외부감사 지적 위험. 51.0 참조.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 12개월 — 자재별·공정별 차이 트렌드를 받아 방향성과 누적치 분석.</li>
 <li><strong>다음 질문</strong>: "지난 12개월 차이 트렌드 자료를 받을 수 있을까요? 방향성이 있나요? 특정 자재군에서 차이가 누적되나요?"</li>
 <li><strong>프로젝트 영향</strong>: 차이 7% 이상 = Material Ledger 검토 필수. 표준원가 재산정 프로젝트 별도.</li>
 </ul></li>

 <li><strong>생산 KPI 5개 — OEE·납기준수·재고회전·불량률·원가율?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 생산팀 운영 성숙도 측정. ADKAR 변화관리 강도 결정.</li>
 <li><strong>진짜 신호</strong>: KPI 5개를 즉답하면 — 성숙한 회사. 답을 못 하면 — 생산팀이 KPI 없이 굴러간다는 신호.</li>
 <li><strong>어떻게 검증하나</strong>: 임원진이 매월 보는 생산 보고서 — 한 부 받아 KPI 5개의 실제 산출 공식과 책임자 확인.</li>
 <li><strong>다음 질문</strong>: "임원진이 매월 보는 생산 보고서 한 부 받을 수 있을까요? KPI 산출 공식이 명문화되어 있나요?"</li>
 <li><strong>프로젝트 영향</strong>: KPI 미정의 = Fiori 대시보드 설계 어려움. 변화관리 추가 4주.</li>
 </ul></li>

 <li><strong>생산계획 주기와 책임자는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: MPS(Master Production Schedule) 단일화 가능성 측정. 계획 충돌 자리 식별.</li>
 <li><strong>진짜 신호</strong>: 책임자 1명이면 정상. 2~3명이 같은 계획을 동시에 짜는 회사가 있다 — 계획 충돌. 컷오버 시 누구의 데이터를 마스터로 가져갈지 충돌.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 1주일 생산계획 변경 이력 — 누가·언제·왜 바꿨는지 추적. 충돌 빈도 측정.</li>
 <li><strong>다음 질문</strong>: "최종 생산계획 확정자는 누구인가요? 그 사람의 결정이 시스템에 자동 반영되나요? 변경 이력은 추적 가능한가요?"</li>
 <li><strong>프로젝트 영향</strong>: 생산계획 단일화 = MPS 안정성. 거버넌스 워크플로 신설.</li>
 </ul></li>

 <li><strong>S&amp;OP 회의는 운영되는가? 어떤 부서가 참여?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: S&amp;OP 정착이 ERP 안정화의 마지막 자리. 회의 표면 운영 vs 실질 운영 구별.</li>
 <li><strong>진짜 신호</strong>: "운영한다"가 답이면 — 회의록 한 부 보여달라. 회의록이 없으면 = 운영 안 됨. 표면만 있는 회사. ERP 도입 후 S&amp;OP 정착이 18개월 추가 과제.</li>
 <li><strong>어떻게 검증하나</strong>: 최근 3개월 회의록을 받아 — 합의된 숫자가 다음 달 실행 데이터와 일치하는지 직접 비교.</li>
 <li><strong>다음 질문</strong>: "최근 3개월 S&amp;OP 회의록을 받을 수 있을까요? 합의된 숫자가 실행으로 이어지나요?"</li>
 <li><strong>프로젝트 영향</strong>: 40장 S&amp;OP 정착이 안정화의 마지막 자리.</li>
 </ul></li>

 <li><strong>Capacity 부족 시 의사결정 — 외주·잔업·납기조정 우선순위는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: CRP 의미 있게 만들 수 있는지·의사결정 룰 명문화 필요성 측정.</li>
 <li><strong>진짜 신호</strong>: 우선순위 명문화 안 됨 = 매번 즉흥 결정. CRP 의미 없음.</li>
 <li><strong>어떻게 검증하나</strong>: 지난 분기 Capacity 부족 사례 1~2건의 — 의사결정 과정·관련자·소요 시간을 추적.</li>
 <li><strong>다음 질문</strong>: "지난 분기 Capacity 부족 사례를 한 두 건 들어볼 수 있을까요? 어떻게 결정하셨나요?"</li>
 <li><strong>프로젝트 영향</strong>: 의사결정 룰 명문화 = 워크플로 신설.</li>
 </ul></li>

 <li><strong>긴급 작업오더(Hot Order) 처리 빈도와 우선순위?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 표준 프로세스 정합성 측정. Hot Order 비중이 높으면 ERP 도입 후에도 같은 패턴 반복.</li>
 <li><strong>진짜 신호</strong>: Hot Order 비중 20% 이상 = 표준 프로세스가 깨진 회사. ERP로 가도 같은 패턴 반복.</li>
 <li><strong>어떻게 검증하나</strong>: Hot Order 100건 — 발생 원인 카테고리(영업 약속 / 자재 결품 / 설계 변경 / 품질 사고)로 분류해 비중 측정.</li>
 <li><strong>다음 질문</strong>: "Hot Order가 발생하는 진짜 원인이 영업의 무리한 약속인가요, 자재 결품인가요, 설계 변경인가요?"</li>
 <li><strong>프로젝트 영향</strong>: Hot Order 원인 식별 = 변화관리 ADKAR 강화 자료.</li>
 </ul></li>

 <li><strong>PLM 시스템 보유 여부, 도면관리 방식은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: PLM-ERP 인터페이스 견적·도면 정합성 측정.</li>
 <li><strong>진짜 신호</strong>: PLM 미보유 + CAD 다종 = 도면 정합성 70% 이하. PLM 도입 권고.</li>
 <li><strong>어떻게 검증하나</strong>: 도면 1장의 최신 Rev 확인까지 — 사용자 5명에게 동시에 부탁해 평균 소요 시간 측정.</li>
 <li><strong>다음 질문</strong>: "도면 한 장의 최신 Rev를 확인하는 데 평균 몇 분 걸리나요? 누가 권한을 가지고 있나요?"</li>
 <li><strong>프로젝트 영향</strong>: PLM 신규 도입 = 별도 견적 5~15억.</li>
 </ul></li>

 <li><strong>품질검사와 생산오더의 연계는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: QM-PP 통합 시나리오 결정. 부적합 자재의 양산 투입 차단 가능성.</li>
 <li><strong>진짜 신호</strong>: 입고 검사가 자동 생성 안 됨 = QM-PP 통합 미운영.</li>
 <li><strong>어떻게 검증하나</strong>: 입고 → 검사 → 양산 투입 — 3단계가 시스템에서 자동 연결되는지 화면으로 직접 확인. 부적합 시 차단 화면 시연 요청.</li>
 <li><strong>다음 질문</strong>: "검사 결과가 생산오더 진행에 자동 반영되나요? 부적합 시 자동 차단되나요? 차단 우회 권한자는 누구인가요?"</li>
 <li><strong>프로젝트 영향</strong>: QM-PP 통합 = 부적합 자재의 양산 투입 차단. 품질 사고 60% 감소.</li>
 </ul></li>

 <li><strong>현재 시스템에서 가장 답답한 5가지는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 새 시스템 가치 제안의 기초·UAT Sign-off 핵심 기준.</li>
 <li><strong>진짜 신호</strong>: 5가지 답이 추상적이면 — 사용자가 시스템에 적응한 상태. 답이 구체적이면 — 진짜 페인 포인트.</li>
 <li><strong>어떻게 검증하나</strong>: 답이 추상적이면 — "그 답답함이 매일 발생하나요? 매주? 매월?"로 빈도를 끌어낸다. 빈도 + 소요 시간 = 자동화 ROI 산정.</li>
 <li><strong>다음 질문</strong>: "그 답답함이 매일 발생하나요? 매주? 매월? 한 번 발생하면 몇 시간 잡아먹나요?"</li>
 <li><strong>프로젝트 영향</strong>: 페인 포인트 명문화 = UAT Sign-off 기준 자동 생성.</li>
 </ul></li>

 <li><strong>새 시스템에서 반드시 되어야 하는 3가지는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: UAT Sign-off의 핵심 기준이 됨. 사용자 성숙도 측정.</li>
 <li><strong>진짜 신호</strong>: 3가지 답이 시스템적 답(예: "Fiori 대시보드") vs 업무적 답(예: "재고 정확도") — 비율로 사용자 성숙도 측정.</li>
 <li><strong>어떻게 검증하나</strong>: 3가지 답을 받은 후 — 각 항목의 측정 가능한 KPI를 같이 정의. KPI 없는 답은 UAT 기준 부적합.</li>
 <li><strong>다음 질문</strong>: "그 3가지가 됐다는 걸 어떻게 측정하시겠어요? KPI 한 개씩 같이 정의해 보실래요?"</li>
 <li><strong>프로젝트 영향</strong>: UAT Sign-off 기준 = 사용자 합의 + KPI 측정 가능성.</li>
 </ul></li>

 <li><strong>현재 가장 시간 많이 잡아먹는 수작업 5가지는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 자동화 ROI의 가장 직접적 산정 근거. 시간당 인건비 × 자동화율 × 12개월 = 연간 절감액.</li>
 <li><strong>진짜 신호</strong>: 수작업 5가지가 — 매일·매주·매월 빈도로 답하면 ROI 즉시 산정 가능.</li>
 <li><strong>어떻게 검증하나</strong>: 각 수작업에 — 담당자·소요 시간·빈도를 받아 연간 인시(Man-hour) 환산.</li>
 <li><strong>다음 질문</strong>: "이 수작업을 안 하면 그 시간으로 무엇을 하시겠습니까?"</li>
 <li><strong>프로젝트 영향</strong>: 수작업 5건 자동화 = 연간 절감 추정 산정.</li>
 </ul></li>

 <li><strong>월말 결산 시 생산팀이 회계팀에 제출하는 자료 목록은?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: PP-FI/CO 인터페이스 자동화 ROI 산정. 회계팀 야근 60% 감소의 직접 자료.</li>
 <li><strong>진짜 신호</strong>: 자료 5건 이상 = 자동화 인터페이스 신설 ROI 분명.</li>
 <li><strong>어떻게 검증하나</strong>: 자료별로 — 작성자·소요 시간·정합성 검증 방식 확인. 엑셀로 작성된 자료 비중이 자동화 우선순위.</li>
 <li><strong>다음 질문</strong>: "이 자료들 중 회계팀이 ERP에서 직접 뽑을 수 있게 되면 가장 가치 있는 게 어느 것인가요?"</li>
 <li><strong>프로젝트 영향</strong>: 자료 자동화 = 결산 D+5 가능성 + 회계팀 야근 감소.</li>
 </ul></li>

 <li><strong>생산팀 인원 구성과 시스템 사용자 수는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 라이선스 산정 + Key User 양성 인원 결정.</li>
 <li><strong>진짜 신호</strong>: 사용자 수 vs 인원 비율 — 50% 이하면 시스템 활용도 낮음. 80% 이상이면 모바일·Fiori 도입 ROI 큼.</li>
 <li><strong>어떻게 검증하나</strong>: 직군별로 — 시스템 사용 빈도(매일·매주·드물게)를 분류.</li>
 <li><strong>다음 질문</strong>: "현장 작업자 중 시스템을 직접 쓰는 비율은 얼마인가요? 모바일 화면이 필요한 직군은요?"</li>
 <li><strong>프로젝트 영향</strong>: 라이선스 + Fiori 모바일 화면 견적 산정.</li>
 </ul></li>

 <li><strong>베트남공장 생산방식이 한국과 다른 점 3가지는?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 본사-자회사 통합 거버넌스의 핵심. 31장 베트남 출장 합의 헌장 5조항의 입력값.</li>
 <li><strong>진짜 신호</strong>: 차이 3가지 답이 — 룰·시스템·문화로 분류 가능하면 통합 설계 명확. 답이 모호하면 추가 인터뷰 필수.</li>
 <li><strong>어떻게 검증하나</strong>: 베트남 법인장과 별도 인터뷰 — 한국 본사 답과 비교해 차이 항목 매트릭스 작성.</li>
 <li><strong>다음 질문</strong>: "베트남 법인장님과 같은 질문을 따로 드려도 될까요? 답이 다른 자리가 진짜 차이일 가능성이 큽니다."</li>
 <li><strong>프로젝트 영향</strong>: 차이 5건 이상 = 룰 별도 설계 4~6주.</li>
 </ul></li>

 <li><strong>천안공장 MES 운영 방식과 만족도?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: MES-ERP 통합 시 MES 측 저항 강도 측정.</li>
 <li><strong>진짜 신호</strong>: 만족도 낮으면 = MES-ERP 통합 시 MES 측 저항. 변화관리 강화.</li>
 <li><strong>어떻게 검증하나</strong>: MES 사용자 5명에게 — 1~10 만족도 점수와 가장 답답한 3가지를 익명으로 받는다.</li>
 <li><strong>다음 질문</strong>: "MES 사용자 5명에게 익명 설문을 돌려도 될까요? 표면 답과 실제 만족도가 다른 경우가 많아서요."</li>
 <li><strong>프로젝트 영향</strong>: MES 저항 강도 = 변화관리 ADKAR 4주 추가.</li>
 </ul></li>

 <li><strong>혹시 빼먹은 질문이 있을까요?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 100개 질문보다 깊은 한 마디가 자주 나오는 자리.</li>
 <li><strong>진짜 신호</strong>: 사용자가 던지는 한 마디가 — 인터뷰 30분의 결론을 뒤집을 때가 있다.</li>
 <li><strong>어떻게 검증하나</strong>: 침묵을 견딘다. 5초 침묵 후 사용자가 던지는 답이 진짜 답.</li>
 <li><strong>다음 질문</strong>: "방금 그 말씀, 한 번 더 자세히 들려주실 수 있을까요?"</li>
 </ul></li>

 <li><strong>이 주제로 누구를 더 만나야 할까요?</strong>
 <ul>
 <li><strong>왜 묻는가</strong>: 추가 인터뷰 대상자 명단 확보 + 인물 간 관계도 파악.</li>
 <li><strong>진짜 신호</strong>: 추천 인물이 — 직속 부하 1명 + 타 부서 1명 + 외부 1명 3축으로 나오면 인터뷰 망 안정.</li>
 <li><strong>어떻게 검증하나</strong>: 추천받은 사람 각각에게 — 같은 사람을 만나도 좋겠냐고 물어 교차 검증.</li>
 <li><strong>다음 질문</strong>: "그분께 제 이름으로 연락드려도 될까요? 어느 채널이 가장 좋을까요?"</li>
 </ul></li>
