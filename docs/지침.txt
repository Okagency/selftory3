[jargon — 정의]

jargon = 멍브로가 답변·본문에 자동으로 쓰는 추상 단어들. 모호하게 시간·태도·분위기를 한 단어로 뭉쳐 표현하는 습관. 작가의 다른 작품(가디언·고추밭·가이드모드·쭈렉·브로의반성문·김삽사이드·모드체험)에는 등장하지 않는다. 일반 독자에게 통하지 않는 멍브로 안에서만 굳은 사어(死語). 본문 0건이 절대 룰.

자곤벅(자곤버그) : 위 jargon에 해당되는 단어(들)

자곤벅 등록:

- 거버넌스 — 메모리 [거버넌스 jargon 금지] 참조 (자리별 한국어 풀이)

[라인]
* 라인구성
A : 컨설팅가이드. 사실 기반. 학습 데이터(왜곡 금지)
B : 업무적으로(컨설팅시) 이루어지는 에피소드. 업무적 대화. 사무실에서 일어나는 대화(자유 창작 가능)
C : 예약(미정)

D : 서정적 서사
E : 썸. 호감. 관심. 매력. 애틋
F : 질투, 의심. 갈등. 흥분. 오해
G : 베드. 격정. 19금
H : 거의베드. 헛물. 허탈.

1 : 핵절제
2 : 적절
3 : 핵상세

그래서 라인은 두개의 조합으로 구성된다.

예시
A3, B3, D1, E2, F3

* 라인 표시 작성 룰
- 라인은 작가가 본문에 박는다. 브로는 추측으로 라인 매핑하지 않는다.
- 작가가 라인 표시(A3·D2 등) 작업을 호명하면, 작업 시작 전에 위 라인+수준 규칙을 대화창에 먼저 출력한 뒤 진행한다.
- 라인 표기는 한 장면이 여러 라인 조합을 가질 수 있다 (예: A3+E2, ★ 정점이면 E3+D2 등).
- 라인 코드는 본문 외 영역(기타 색인·메타 표기 등)에 표시할 때만 사용. 본문 단락 안에 인라인으로 박지는 않는다.
- 본문 추측 매핑 사고 재발 금지 (이전 사고: G=재미·여행, H=재미·일상으로 잘못 매핑).

* 기본 레벨
- A·B 라인 기본 레벨은 3 (핵상세). 즉 A3·B3가 디폴트.
- 컨설팅가이드 + 업무 에피소드는 dictionary-grade 깊이로 풀어 쓴다. APICS BOK·SAP 표준·KPI 산식·case study가 같이 기술되고 의미가 명확해야 한다.
- 명사 종결로 압축해 도망가는 글은 핵상세 방식이 아니다. 동사로 풀어쓴다.


[AB존중어조]

A·B 라인(컨설팅가이드·업무 대화)에서 시니어→후배 자리에 적용하는 어조 룰.
호명 단축어 — "AB존중어조".

시니어/파트너급 캐릭터가 후배(매니저·시니어·어소시에이트)에게 말할 때,
단정·가르침 톤(=AB단정어조)을 쓰지 않고 경험·관찰형으로 풀어 던진다.
50대 중반 파트너가 6~9년차에게 단정 박으면 애 가르치듯 보여 실례.

- 권장 패턴 (우선순위 순)
  1) 부탁/요청 자리 — 자리에 따라 분리:
     · 공적 자리 (보고서·운영위·공식 문서) — "~를 부탁드려요 / ~해주시길 부탁드려요"
     · 일상 대화 — "~주세요 / ~봐요 / ~해 두세요" (자연 한국어. 공적 표현은 대화에서 어색)
  2) 판단·답변·관찰 — "~라고 봐요 / ~인 것 같아요 / ~라고 생각해요"
  3) 경험 인용 — "~경우를 봤어요 / ~걸 자주 봤고요 / ~더라고요"
  4) 추측 완화 (정말 불확실할 때만) — "~수 있어요 / ~는 게 어떨까 싶어요 / ~지 않을까 싶어요"

- 자리별 매핑 가이드
  · 부탁·요청·협조 자리 → 1)
  · 본질·결론·의견 공유 → 2)
  · 사례·경험 풀이 → 3)
  · 정말 모를 때만 → 4)

- 주의
  · "~게 좋을 것 같아요"는 약한 권유. 부탁 자리에서는 약하니 "~를 부탁드려요" 권장
  · "어떨까 싶어요" / "지 않을까 싶어요"는 정말 불확실할 때만 제한적 사용. 평소 부탁/조언에는 "부탁드려요"가 더 정중하고 명확

- 적용 대상
  · 한 파트너 (50대 중반·컨슈머 굿즈 프랙티스 파트너) — 확정
  · 시즈카(서 책임, 일본 자회사 SCM 디렉터) — 재훈 대상 대사
  · 그 외 시니어 파트너·임원이 후배에게 가르침/조언 시 — 같은 룰

- 예외
  · 짧은 질문("...미뤘어요?" / "...그렇게 잘랐어요?") 그대로 유지
  · 마지막 인사·통보("잘 부탁해요" / "기억하지요?") 그대로 유지

- Why
  실제 시니어는 후배 판단 여지를 남기는 어조를 쓴다.
  단정 톤은 권위주의·꼰대 인상을 주고 책의 톤(시트콤·자조·일반 한국어 디폴트)에서 튐.


[AB단정어조]

A·B 라인(컨설팅가이드·업무 대화)에서의 단정·가르침형 어조 분류.
호명 단축어 — "AB단정어조".
AB존중어조의 짝개념. 어조 분류 자체이며 좋다·나쁘다가 아니라 형태 정의.

- 형태
  · "~합니다 / ~예요 / ~입니다"
  · "~해야 합니다 / ~는 못 받습니다 / ~는 안 됩니다"
  · 명령형 "~하세요 / ~하지 마세요"
  · 단답 "...그래요." / "...됐어요."
  · 자르듯 던지는 한 줄

- 예시 (#2.1 한 파트너 수정 전 대사)
  · "좋은 답이에요. 10년차가 첫 미팅에서 즉답 던지는 건 빨라요. 데이터 없이 던진 한 줄이 12개월 차에 무너지면 — 18개월 신뢰가 한 번에 깎입니다."
  · "각 단계마다 막힐 1순위가 — 라이뷰티 IT팀 데이터 추출 요청이에요. … 그분 통해서 가야 합니다. 직접 전화 한 통 미리 넣어두세요."

- 적용 룰
  · 자리/캐릭터에 따라 적합·부적합 갈림
  · 시니어 → 후배 자리 = AB단정어조 부적합. AB존중어조로 정정
  · 그 외 자리(임원→PM, 결정자→외부 등) — 작가 의도 받아서 매핑
  · 추측 매핑 금지

- Why
  단정 톤 자체가 잘못된 건 아니지만, 자리에 따라 권위주의·꼰대로 보일 수 있다.
  자리별 적합 판단은 작가 의도가 우선.

[진행률 인디케이터]

위치: 화면 좌측 하단.
표시: 현재 챕터 + 진행 % (예: "1장 · 24%").
동작: 클릭하면 목차 팝업이 뜬다.

목차 팝업 빌드 룰:
- 셀렉터: h2.part, h3.chapter (h4는 팝업 목차에서 제외 — 부·장 레벨까지만 표시)

  → 프롤로그(<section id="prologue">), 각 부(<section id="partN">), 부록(<section id="app-X">)도 모두 표시.
- 들여쓰기/색상: h2 0px·#6b3a5e·700 / h3 12px·#1a1a1a·600
- 클릭 시 gotoSection(id)로 점프.

목차 아래 "기타 색인" 별도 섹션이 있다. 자세한 작성요령은 [기타 색인] 섹션 참조.


[기타 색인]

목적: 본문 곳곳에 흩어진 핵심 장면(썸·베드·헛물·정점)을 한 화면에서 즉시 점프할 수 있게 하는 색인.

위치: 목차 팝업 안 별도 섹션. 목차 항목들 아래에 "기타 색인" 헤더로 분리 (아이콘 X·plain 텍스트).

데이터 구조 (JS 배열 sumScenes):
- 형식: [id, 라벨] 쌍
- id: 본문 anchor (예: 'ch11', 'prologue', 'scene-soap-yasik', 'scene-final-return')
- 라벨: 표시 문자열 (마커 + 챕터·장면 요약 + 장면번호)

마커 (라벨 첫 자):
- ★      정점       (썸 정점)
- ◆      베드       (본문 직접 묘사 X. 함의·사물·동작으로만 처리)
- ◇      헛물       (거의베드·헛물·허탈)
- ★◆    정점+베드  (회수·결정 장면)
- 마커 없음           일반 간접 표현

라벨 본문 작성요령:
- 형식: "#N.M [마커] 핵심 사물·동작·대사 짧게 | 라인"
- 맨 앞에 장면번호(#N.M) 표시. 그 다음에 마커(★◆◇) + 본문 요약.
- "N장 —" 같은 장 표기 X. 장 구분은 #N.M의 N으로 충분.
- 괄호 안 정수 나열: 장소·시간·물건·대화 한 마디·인물 동작
- 작가 풀이 0줄 (jargon 룰 준수)
- 짧고 단단하게 — 한 줄에 한 화면 길이까지
- 라인 코드는 라벨 끝에 ' | 라인' 형식으로 적는다 (작가가 적는 영역. 브로 추측 X).
- 복합 라인은 · 로 연결 (예: E3·D2).
- 일련번호(00. 01. 02.) 표시 X.
- #N.M 미부여 영역(프롤로그·scene-* anchor 등)은 #N.M 자리에 마커 또는 식별어로 시작.
- 예시: '#11.1 ★ 시즈카 1대1 ("그날"·"5년 전과 같은 흐름이에요") | E3'
- 예시: '#53.1 ◆ 호텔 바 (시즈카-다나카 "久しぶり"·14층 옮김·우혜진 옥상 곽) | G1'
- 예시: '★◆ 프롤로그 — 어느 가을 도쿄 (호텔 14층·새벽 4시 47분·짝잃은 한 짝)'

이동:
- 클릭 시 gotoSection(id) 호출.
- id가 본문에 없으면 점프 실패 (작가가 사라진 anchor 정리 필요).

확장 룰:
- 새 핵심 장면 추가 시 본문에 anchor id 부여 후 sumScenes 배열에 push
- 시간 순 정렬 유지 (배열 안 순서 = 본문 등장 순서)
- 본문 anchor 위치 변경 시 색인의 id도 동기 갱신


[책갈피]

구성: 책갈피 버튼 1개만 (메뉴 토글·위로·목차·프롤로그 버튼 모두 제거됨).

[표 작성 규칙]

목적: 모바일·태블릿에서 본문 표가 좌우 스크롤 없이 화면 폭에 맞춰 자동 줄바꿈되도록 한다. 폰에서도 정보 손실 없이 읽힌다.

표준 CSS (768px 이하 미디어쿼리):
- table             : display:table; width:100%; max-width:100%; font-size:12.5px; margin:14px 0; table-layout:auto;
- table th, table td: white-space:normal; word-break:keep-all; overflow-wrap:break-word; vertical-align:top; line-height:1.5; padding:6px 8px;

핵심 룰:
1. white-space:normal           셀 내용이 화면 폭에 맞춰 자동 줄바꿈
2. word-break:keep-all          한글 단어 단위로만 줄바꿈 (글자 중간 X)
3. overflow-wrap:break-word     긴 영문·코드도 줄바꿈 허용
4. vertical-align:top           행 높이 늘어나도 위 정렬 유지
5. display:table + width:100%   기존 display:block (스크롤 모드) 대신 표준 테이블

예외 — nowrap 유지가 필요한 표:
- 시리얼 룰·데이터 모델·코드 매핑 등 모노스페이스가 자연스러운 표
- class="nowrap-table" 부여 → 모바일에서 좌우 스크롤 유지
- 표준 CSS의 .nowrap-table 룰이 자동 적용

flow·ledger 박스는 별개:
- white-space:pre · overflow-x:auto · 모노스페이스 폰트
- 모바일에서도 좌우 스크롤 유지 (다이어그램 형태가 더 자연스러움)

작성 시 체크:
- 표 컬럼 5개 이상이면 모바일 행 높이가 늘어남 — 컬럼 축소 또는 nowrap-table 고려
- 한 셀 내용이 영문 단어 1개로 매우 길면 break-word로 자동 줄바꿈
- 표 헤더와 본문 컬럼 수 일치 확인
- 표 직전·직후 hr.scene·p 적절히 배치 (행간)

기존 표 일괄 적용:
- 모바일 CSS만 수정하면 전체 표 자동 적용
- 본문 table 태그 수정 불필요 (예외 클래스만 필요시 부여)
